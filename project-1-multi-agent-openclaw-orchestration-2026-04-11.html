Project 1: Multi-Agent OpenClaw Setup and Orchestration

Designing a role-based multi-agent personal operating system in OpenClaw.

One-line summary: I designed and operationalized a multi-agent assistant system in OpenClaw where specialized agents handled coding, research, communications, and memory curation, while a master orchestrator coordinated work, preserved context, and enforced workflow boundaries.

What the project was

This project started from a simple but important problem: a single assistant can do many things, but once work becomes varied, ongoing, and cross-functional, it becomes messy to manage through one undifferentiated agent.

So instead of treating the assistant like one general blob, I structured it into a team of specialized agents:

  • Jeeves as the orchestrator
  • Ada for coding and technical work
  • Hermes for research, operations, and communications
  • Scribe for summaries, organization, and memory curation

The goal was not just to create personas. It was to create a system where delegation, role boundaries, context recovery, and long-term continuity were all intentional parts of the design.

Problem

A monolithic assistant setup creates a few recurring issues:

  • role confusion, where the same agent tries to do research, coding, messaging, and organization all at once
  • weak continuity, where context gets lost between sessions
  • poor specialization, where every task is handled in the same generic style
  • hard-to-audit work, where it is unclear who did what and why
  • brittle delegation, where cross-agent work can drift or disappear without durable state

In practice, this meant that even when the system was capable, it was not yet behaving like a reliable operating model.

What I wanted to achieve

I wanted the system to feel less like chatting with one smart bot and more like a small, well-run staff:

  • one agent coordinates
  • specialists do focused work
  • important state survives resets and handoffs
  • memory is shared where it should be shared
  • external actions stay bounded and deliberate
  • the user does not have to keep repeating preferences or role assignments

What I built

I turned the setup into a role-based multi-agent architecture with explicit workflow rules.

Agent roles

  • Jeeves owns orchestration, synthesis, delegation, and overall system integrity
  • Ada owns coding, implementation, debugging, and technical operations
  • Hermes owns research, communications workflows, and operational follow-through
  • Scribe owns summaries, memory curation, documentation quality, and cleanup

This was not left implicit. The routing policy was made official so the system would treat it as durable operating truth rather than a soft preference.

Delegation and ownership model

  • Jeeves decides how work is routed
  • specialists execute within bounded roles
  • sensitive, destructive, or external actions remain gated
  • cross-agent work gets durable handoff state instead of relying on conversation alone

This created a clearer division between can do and should do.

Shared-state structure

I used the shared workspace as the system’s durable memory layer:

  • daily logs
  • reminder ledger
  • structured project memory
  • handoff files
  • preference notes
  • project-specific notes

This made the agents less dependent on fragile session memory and more dependent on explicit, recoverable artifacts.

Handoff discipline

For non-trivial cross-agent tasks, baton files were introduced so work could survive resets and role changes. That made delegation auditable and reduced the chance of work quietly disappearing between specialists.

Technical and system details

1. Routing policy became explicit

A common failure mode in agent systems is that role definitions exist in theory but not in behavior. To fix that, the routing model was documented and normalized so the system consistently treated research and ops as Hermes work, coding as Ada work, curation as Scribe work, and orchestration as Jeeves work.

2. Durable memory replaced vague continuity

Instead of assuming the system would remember, I leaned on file-backed continuity:

  • reminders written to a canonical ledger
  • project notes stored in recoverable locations
  • policy decisions written into durable memory
  • baton files used for cross-session delegated work

This turned continuity from an illusion into infrastructure.

3. Shared vs local artifact boundaries were clarified

One major insight was that shared canonical artifacts should live in the shared workspace, but specialists also need startup-visible pointers or doctrine that makes those resources discoverable from their own context. Without that, good memory can exist and still be practically invisible.

4. Bounded autonomy was enforced

Specialists were allowed to analyze, draft, stage, verify, and recommend, but not silently cross external or risky boundaries just because they technically could. That matters a lot in a real assistant system.

Challenges

Challenge 1: Specialists were not reliably retrieving the right policy

Even with memory written down, the specialists sometimes answered from stale role priors or old identity assumptions instead of the current official rules.

Challenge 2: Startup doctrine mattered more than expected

The issue was not only memory exists or not. It was whether the right files were visible and loaded early enough for each specialist. Missing local startup doctrine and conflicting role definitions created retrieval drift.

Challenge 3: Shared artifacts were not enough by themselves

A canonical file in a shared workspace does not help much if specialists do not have an explicit discovery path to it.

Challenge 4: Session state could mask fixes

After repairs, live sessions could still behave as if old rules were in force, which made validation tricky. Fresh-session testing turned out to be necessary after doctrine changes.

Learnings

1. Systems fail at the seams

The hard part is often not raw capability. It is routing, startup visibility, artifact discoverability, handoff discipline, and persistence across sessions.

2. Memory must be retrievable

A system can have the correct information on disk and still fail if the retrieval path is weak or inconsistent.

3. Role design needs enforcement

Specialist roles are not real until they are encoded in doctrine, discoverable at startup, reinforced by workflow, and tested against real tasks.

4. Durable files beat implied memory

The more I treated reminders, policies, and handoffs as explicit artifacts, the more reliable the whole system became.

Good orchestration turned out to be a product decision, not just prompt tuning. It involved deciding who does what, what should persist, what should escalate, what should be recoverable, and what the user should never need to repeat.

Why this project matters

This project is important because it shows I was not just using an AI tool casually. I was designing an operating model for AI agents.

It demonstrates:

  • systems thinking
  • workflow design
  • role architecture
  • memory-aware product design
  • practical understanding of agent limitations
  • the ability to improve reliability through structure, not just model choice
