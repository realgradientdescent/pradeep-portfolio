Case Study

Building a Simulation Platform for High-Stakes Internal Communications

This project began as a lightweight script for testing employee reactions to management messages and evolved into a multi-component product: a weighted simulation engine, a multi-provider LLM layer, a Monte Carlo scenario model, a React dashboard, durable async job orchestration, saved-run retrieval, and automated executive reporting in PDF and PPTX formats.

The most important lesson from this project was that decision-support systems are not just about modeling the right answer. They are about reliability, recoverability, interpretability, and packaging the output in a form leaders can actually use.

Quick Snapshot

  • Independent end-to-end build spanning backend, frontend, AI integration, reporting, and product design.
  • 60 weighted personas representing a 450-person IT organization.
  • Supports heuristic runs, LLM-backed runs, and Monte Carlo scenario analysis.
  • Includes executive and board-ready report generation in Markdown, HTML, PDF, and PPTX formats.
  • Built to solve a realistic leadership problem: how a message sounds from the top versus how it lands across the org.

Project Framing

The Problem I Was Solving

Leadership communications often fail not because the message is empty, but because different audiences infer different intent, fairness, and personal risk from the same text. I wanted to build a tool that could simulate those differences before a message was sent, then turn the output into something that managers and executives could act on.

Personas 60 Expanded from an initial 15-archetype prototype into a broader weighted org model.
Represented Org 450 Weighted across employees, contractors, geography, leaders, and principal architects.
Delivery Modes 3 Heuristic simulation, LLM-backed simulation, and Monte Carlo scenario analysis.
Output Surfaces 4 Dashboard, JSON/Markdown runs, PDF/HTML briefs, and editable PowerPoint decks.

Technology

Stack and Responsibilities

The stack was deliberately lightweight where possible and more specialized only where the problem demanded it. The system relies heavily on plain Python and filesystem-based persistence, with React used for the primary interactive surface and Node introduced only for PowerPoint generation.

Layer Python for simulation logic, orchestration, file output, LLM transport, and report generation.
Backend Standard library HTTP server plus threads in dashboard_server.py for APIs, async job handling, static asset serving, saved-run loading, and persistent job status.
Frontend React 19 + Vite 8 + vanilla CSS for the dashboard UI, run setup, polling, visual summaries, Monte Carlo explanation, and persona exploration.
Simulation Weighted persona modeling in simulate.py, including heuristic scoring, richer reactions, stochastic adjustments, Monte Carlo ranges, and checkpointed outputs.
AI Layer Groq, DeepSeek, OpenAI, and Ollama behind a shared transport and schema layer in llm_client.py, including model defaults, retries, API-key readiness checks, and usage/cost accounting.
Persistence Filesystem-first architecture: JSON and Markdown run outputs in outputs/, job state and logs in outputs/_jobs, selector JSONs for reports, and local provider config files.
Reporting Python-generated Markdown and HTML, PDF rendering through headless Edge/Chrome with fallback generation, and editable slide decks via PptxGenJS 3.12.
Data Model JSON-based personas and messages, representing 270 employees, 180 contractors, 315 US-based staff, 135 India-based staff, 25 people leaders, and 15 principal architects.

Architecture

How the System Works End to End

Architecturally, the project moved from a single script into a layered system with a simulation engine, a provider abstraction, a dashboard backend, a React frontend, and a reporting pipeline. The key design move was treating saved outputs and in-flight jobs as durable product objects rather than temporary implementation details.

Design Choices

Important Decisions That Shaped the Product

Several design decisions had outsized impact on the usefulness and resilience of the final system. These are worth highlighting in a portfolio because they show judgment, not just implementation volume.

Weighted Personas Instead of Flat Personas

  • Expanded the model from 15 generic personas to 60 weighted personas.
  • Represented real organizational proportions: employees vs. contractors, US vs. India, leaders vs. ICs, architects vs. others.
  • Allowed summary metrics to represent a simulated org rather than a collection of disconnected characters.

Keep Heuristic Mode Even After Adding LLMs

  • Preserved a low-cost, fast simulation path for shaping drafts quickly.
  • Used LLM mode when realism and richer language mattered more than speed.
  • Created a more practical product because not every run needs hosted inference.

Treat Saved Outputs as First-Class Product Objects

  • Built run listing, retrieval, and reloading into the experience instead of treating outputs as throwaway files.
  • Enabled executive reporting to operate off saved runs, not live state.
  • Made the tool more credible as a decision-support system.

The most consequential architectural shift was moving from "one request, one response" thinking to "long-running jobs with durable lifecycle state." Once Monte Carlo, LLM calls, refresh behavior, and background tasks entered the picture, durability and observability became product requirements, not backend extras.

Execution Reality

Major Issues Faced and How They Were Resolved

This is where the project shows the most systems thinking. The interesting work was not only in building features, but in responding to failures, redesigning assumptions, and hardening the product as real-world conditions surfaced.

1. The System Outgrew the Original Single-Script Mental Model

Symptoms

The project started simple, but once LLMs, Monte Carlo runs, dashboards, saved runs, and reporting were added, logic began to span too many concerns.

Root Cause

The original architecture assumed short-lived runs and a mostly local script workflow, not a product with interactive state and historical records.

Resolution

Separated concerns into the simulation engine, provider layer, dashboard server, run repository, React UI, and reporting generators.

Why It Matters

This shows the ability to recognize when a prototype architecture is no longer sufficient and evolve it without losing momentum.

2. Provider Configuration Was Too Fragile for Real Reuse

Symptoms

Dashboard runs would say provider keys were missing depending on which terminal launched the backend and whether environment variables were set there.

Root Cause

Configuration was initially modeled as shell-session state rather than part of the product's durable operating setup.

Resolution

Added local config support through provider secrets files, provider readiness indicators in the UI, model hints, and clear precedence rules.

Why It Matters

This is a systems-thinking move because it turns "works on my shell" into "works as an operating model."

3. Large LLM Runs Surfaced Schema, Rate-Limit, and Reliability Problems

Symptoms

Groq hit rate limits, some model responses failed JSON or tool schema validation, and provider-specific quirks caused otherwise valid runs to fail.

Root Cause

At larger scale, the system moved from ideal prompt paths into provider-specific operational behavior: TPM ceilings, schema drift, and inconsistent output fidelity.

Resolution

Added backoff for 429s, transport retries, persona caps for fast validation, narrower schemas, model guidance, pricing metadata, and lighter default models.

Why It Matters

It demonstrates an understanding that AI systems succeed through operational discipline, not prompts alone.

4. The Dashboard Initially Lost State on Refresh

Symptoms

Refreshes could reset message text, mode, provider, Monte Carlo settings, and active run context. Users returning later could see stale or misleading state.

Root Cause

The original frontend treated the dashboard as a page, not a stateful task workspace with authored form state and active background jobs.

Resolution

Persisted form state, filter state, viewed-run state, active job ids, and request snapshots. Added job polling and reconnect behavior after refresh.

Why It Matters

This shows product empathy: the implementation changed to fit the user's actual workflow, not the developer's happy path.

5. Long-Running Jobs Needed Durable Logs, Checkpoints, and Honest Failure States

Symptoms

Runs could fail after stepping away, come back as "loaded" without results, or disappear without enough evidence to diagnose what happened.

Root Cause

Async jobs originally relied too much on in-memory state and final-save assumptions. There was not enough durable evidence or intermediate persistence.

Resolution

Added persisted job records, log files, checkpointed output files after Monte Carlo passes, explicit interrupted states, recovery from latest checkpoint, and UI activity panels.

Why It Matters

This is classic systems leadership: design for observability and graceful degradation instead of pretending failures will not happen.

6. Statistical Output Was Correct but Not Executive-Friendly

Symptoms

Percentile language like p10/p50/p90 was mathematically clear but not intuitive for leaders using the dashboard as a decision-support tool.

Root Cause

The system initially optimized for correctness over executive readability.

Resolution

Reframed Monte Carlo output into low case, typical case, and high case, added explanatory copy, and turned thresholds into direct business questions.

Why It Matters

This shows the ability to translate analytics into decisions, which is often what separates useful tools from clever tools.

7. Message Scope and Segment Logic Needed Tight Alignment

Symptoms

Some executive summary and watchlist text referenced contractors even when the run was explicitly filtered to employees only.

Root Cause

Some summary logic was still reading the broader result shape rather than the filtered result set shown to the user.

Resolution

Made segment summaries, watchlists, and narrative generation scope-aware so the executive readout matches the exact filtered population.

Why It Matters

It demonstrates attention to trustworthiness. Leadership tools lose credibility quickly if the narrative does not match the data slice being shown.

8. Network and Provider Errors Needed Operational Hardening

Symptoms

Runs hit connection resets, DNS failures, provider disconnects, and transient transport issues, especially with longer requests and stepped-away sessions.

Root Cause

Hosted LLM runs sit on top of real networks, sleep states, VPNs, and provider edges. These errors were not all retried initially.

Resolution

Expanded retry logic beyond rate limits to include DNS and connection-reset classes of failure, improved dashboard error translation, and surfaced interruption banners clearly.

Why It Matters

This reflects an operator mindset: production-grade behavior requires accounting for network reality, not just application logic.

9. The Automation Layer Exposed an Important Integration Boundary

Symptoms

Some PDF automation runs appeared in Codex as pending then archived, without generating new files or visible output threads.

Root Cause

The automation launcher could fail before a real run thread was created, even though the prompt definition itself was valid.

Resolution

Added direct local generator scripts for executive and board briefs and simplified automation prompts to call those scripts deterministically.

Why It Matters

This shows a practical bias: when a workflow boundary is unreliable, simplify the contract instead of stacking more cleverness on top of it.

10. Repo Hygiene Needed Attention Once the Product Produced Artifacts

Symptoms

The first git snapshot risked including outputs, PDFs, PPTX files, logs, caches, and generated artifacts instead of a clean source tree.

Root Cause

The project had matured into a system that produces many runtime assets, but ignore rules had not grown with it.

Resolution

Expanded ignore rules to exclude outputs, generated reports, caches, local secrets, and temporary review artifacts.

Why It Matters

Strong engineering includes stewardship of the development environment and long-term maintainability, not just feature work.

Leadership Signals

What This Project Demonstrates Beyond Coding

If this project is used in a portfolio, the strongest story is not simply "I built a dashboard." The stronger story is that I built and iterated on a socio-technical decision-support system with attention to product design, operational resilience, and executive usability.

System Thinking

  • Modeled the organization as a weighted system rather than a flat average user.
  • Recognized that reliability, cost, state recovery, and reporting are part of the product, not edge features.
  • Designed for retrieval, checkpointing, and auditability instead of just producing outputs.
  • Built a workflow where simulation, interpretation, and executive packaging are part of the same system.

Leadership Qualities

  • Translated ambiguous business problems into a structured product roadmap.
  • Balanced engineering rigor with stakeholder readability and adoption.
  • Handled setbacks by redesigning assumptions rather than patching symptoms.
  • Documented learnings so the work becomes organizational knowledge, not just a one-off project.
The project demonstrates a pattern I would want to highlight in a portfolio: I can take an ambiguous, cross-functional problem, build the first version quickly, recognize where the original architecture stops being good enough, and then evolve the system into something more resilient, interpretable, and useful.

Reflection

Key Learnings Worth Calling Out Publicly

This section is especially useful if the eventual portfolio site includes a "What I learned" or "What I would do differently" component. Those reflections often communicate maturity better than feature lists.

Reliability Matters as Much as Intelligence

  • Good prompts and structured outputs are necessary but not sufficient.
  • Retries, checkpoints, state persistence, and logs are what make AI features dependable in practice.

Analytics Need Translation

  • Mathematically correct output is not automatically decision-ready output.
  • Leaders need framing, thresholds, and scenario narratives, not raw statistical jargon alone.

Adoption Is a Product Design Problem

  • People use tools that match their workflow, not the developer's assumptions.
  • Refresh recovery, saved runs, and report packaging mattered because they supported real behavior.
A clean way to summarize the learning is: this project taught me how to move from model-centric thinking to system-centric thinking. The hard part was not just generating reactions; it was building the surrounding product so those reactions could be trusted, revisited, explained, and acted on.

Portfolio Translation

How to Turn This Into a Strong Portfolio Page

The best public-facing version of this project should not try to reproduce the entire engineering playbook. Instead, it should use the strongest parts of this document as structure, then let screenshots, diagrams, and a few well-written callouts do most of the work.

Recommended Site Sections

  • Hero: project name, one-sentence problem statement, one short thesis about what makes the project interesting.
  • Snapshot: stack, scope, personas, outputs, and a concise summary of the final product.
  • Architecture: one polished diagram and three or four design decisions.
  • Challenges: a small number of issue-resolution stories that show judgment under real constraints.
  • Learnings: short reflection on reliability, product design, and leadership thinking.
  • Artifacts: dashboard screenshots, example report output, and optionally a short GIF of the workflow.

Recommended Visual Artifacts

  • Dashboard overview with executive summary, Monte Carlo range, and persona explorer.
  • One architecture diagram showing dashboard to API to simulation to outputs.
  • One screenshot of a generated board or executive PDF or PPTX deliverable.
  • A single issue-resolution callout showing how job checkpointing or retry logic improved reliability.
  • Optional: a before-and-after view of the simulator evolving from a script into a full product.

For the public version, do not try to say everything. Lead with the business problem, show one strong architecture diagram, highlight two or three real engineering challenges, and end with what the project taught you about building reliable AI-enabled systems.
